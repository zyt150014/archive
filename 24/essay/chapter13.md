# chapter13 中间人

# 在中心化和去中心化中，这二者的区别就是中间人。
# 也正如先前所说，中间人的存在源自于个体间直接交流的不可信。而物质的交易则是个体之间直接交流的一种重要形式。在直接交易中，也不存在同时的概念。每个完整的交易状态都有许多中间状态，而在任意一个中间状态都有可能导致这场交易的终止。这种终止往往也会导致一方的利益遭受损失，因而在保障自身利益的情况下，许多交易都会引入中间人作为担保。
# 但这种中间人也只是提高了利益不受损的概率，并不能完全避免。而且中间人作为担保，承担了这笔交易的风险，也必定能从中得到回报，这也是双方愿意让出部分利益换取更高可靠性的行为。而如果交易的一方愿意付出额外的时间和精力成本，做出更多的担保，那么在这个情况下这一方也承担了中间人的角色。但这种一方承担中间人的角色的情况大多都是一方在这场交易中付出了更多的成本，但另一方并没有付出更多，却提高了这场交易的可靠性。如果交易双方都承担一部分中间人的角色，并且愿意额外付出一些成本，那么这场交易也很容易推动。
# 但事情总是出乎意料的。如果交易一方的时间成本大于寻找中间人进行代理，或者双方在额外付出的成本上不能达成一致，也有可能是这类交易的前提是有三方介入，都会导致直接交易的不可行。
# 数据信息的交流则更加简单。首先是个人天然具有与他人进行信息交流的意愿，来了解他人的认知看法，这是形成感情的必要条件。更重要的就是数据信息可以进行无限的复制，因而成本近乎为0，同时也能满足个人对于自身和群体之间联系的要求。这种数据信息的交流也往往需要中间人进行传递，以实现不同物质空间位置的高效交流。
# 除了这些交流之外，中间人也有更多的表现形式，现在让我们来看看在完成一件事情时我们需要做些什么。在多个人参与来解决一个问题时，这个多人集体也就是一个问题求解器。将具体的问题以及期望的结果进行输入之后，最终的结果与期望的结果之间的区别多少也就是这个求解器的质量好坏。而在从问题的提出到问题的解决的这个过程中，从上游到下游形成了一条流水线，其中包括各种中间环节。提出问题之后的第一个处理，就是管理者先把问题拆分成几个小问题，然后将这些小问题分配给不同的小集体，并持续追踪小问题的结果以及协调参与不同小问题的人员。每个人则作为中间环节的一部分，只需要对自己负责的环节的输入输出进行质量把控，使其最终能满足整体的要求即可。
# 纯粹的数据处理则是中间人更加有效的场景。map和reduce作为数据处理的两个抽象中间阶段，也是解决问题的一种朴素思想。在数据处理方面，map过程对应着将数据划分成更小的数据，并决定如何去进行划分，reduce过程则是将多个map过程已经处理好的数据进行合并，最终完成所有数据的处理。你可能会问在map和reduce之间应该存在一个中间过程，用来从更小的数据得到这些数据的结果。当然这个中间过程也能再次进行map-reduce过程，直到将最后的处理变成原子操作。在这个地方数据代替了问题，编写的程序代替了个人的负责环节。
# 除此之外，我们创造的各种工具也是我们对物质世界认知的中间人。这些工具将各种形式的数据进行中间处理，转变成我们已经熟悉的类型。而从已经熟悉的数据到在大脑中产生相应的数据概念，也需要各种感受器和神经传导的中间过程。这些都是中间人在不同概念上的存在形式。
# 当然以上都是在某种层面上有着一定意义的中间人，也存在直接转发不做任何处理的中间人，代理就是其中的一种。代理的存在是因为代理的上下游之间尚未建立联系，或者是由于一些关系无法建立联系，而代理作为能接触到二者的中间人，需要在上下游之间使用代理将二者联系在一起。尚未建立联系的现实例子就是信息差，利用这种信息差来赚取差价。而无法建立联系多半则是因为人为限制，将正常渠道进行管控或者直接切断。这种没有自身参与的中间人在能够被去掉之时也很容易被清除来提高系统效率。
# 中间人参与中间环节，也只对自身的输入输出负责。
